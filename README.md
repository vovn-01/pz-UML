# Practical lesson pz-UML
Побудова повердінкових UML-діаграм для проєктування інформаційних систем

## Тема - Система автоматизованого обліку та контролю строків носіння форменого одягу військовослужбовців 

## 1. Діаграма варіантів використання (Use Case Diagram)

Ця діаграма відображає основні ролі (акторів) та їхні функціональні можливості в системі.

**Призначення:** Визначити межі системи та взаємодію користувачів із її функціями.

`code`

    useCaseDiagram
    actor "Відповідальна особа (Речовик)" as Admin
    actor "Військовослужбовець" as Soldier
    actor "Система (Таймер)" as SystemClock

    package "Система обліку форменого одягу" {
        usecase "Внесення даних про видачу майна" as UC1
        usecase "Перегляд залишку терміну носіння" as UC2
        usecase "Формування звіту про потребу заміни" as UC3
        usecase "Автоматичне сповіщення про закінчення строку" as UC4
        usecase "Списання зношеного майна" as UC5
    }

    Admin --> UC1
    Admin --> UC3
    Admin --> UC5
    Soldier --> UC2
    SystemClock --> UC4
    UC4 ..> UC3 : <<include>>

`code`

## 2. Діаграма послідовності (Sequence Diagram)

Діаграма послідовності ілюструє життєвий цикл запиту на сканування.

Діаграма ілюструє процес видачі нового однострою та ініціалізацію контролю термінів.

**Призначення:** Показати логіку взаємодії між об'єктами системи у часі.

`code`

    sequenceDiagram
    autonumber
    actor Admin as Речовик
    participant Sys as Інтерфейс Системи
    participant DB as База Даних
    participant Notif as Сервіс Сповіщень

    Admin->>Sys: Вводить ПІБ бійця та тип майна
    Sys->>DB: Запит на перевірку норм забезпечення
    DB-->>Sys: Норми отримано (напр. 12 міс.)
    
    Admin->>Sys: Підтверджує видачу (Дата: сьогодні)
    Sys->>DB: Зберегти запис про видачу та дату експлуатації
    DB-->>Sys: Запис успішний
    
    Sys->>Sys: Розрахунок дати закінчення терміну
    Sys->>Notif: Запланувати сповіщення на [Дата]
    
    Sys-->>Admin: Операція завершена, картку оновлено
`code`

## 3. Діаграма діяльності (Activity Diagram)

Діаграма описує бізнес-процес контролю термінів: від перевірки стану до отримання нового майна.

**Призначення:** Відобразити алгоритм роботи системи та логічні розгалуження.

`code`

    stateDiagram-v2
    [*] --> ПеревіркаТермінів: Щоденний моніторинг
    
    state ПеревіркаТермінів {
        direction LR
        [*] --> ПорівнянняДат
        ПорівнянняДат --> ВизначенняСтатусу
    }
    
    ПеревіркаТермінів --> ТермінВийшов: ДатаПоточна >= ДатаГранична
    ПеревіркаТермінів --> ВЕксплуатації: Термін ще триває
    
    ВЕксплуатації --> ПеревіркаТермінів: Наступний цикл
    
    ТермінВийшов --> ФормуванняЗаявки: Автоматично
    ФормуванняЗаявки --> ПогодженняКерівництвом
    
    state ПогодженняКерівництвом <<choice>>
    ПогодженняКерівництвом --> ВидачаМайна: Схвалено
    ПогодженняКерівництвом --> Відмова: Відхилено (напр. втрата)
    
    ВидачаМайна --> СписанняСтарого: Оновлення інвентарної картки
    СписанняСтарого --> [*]
`code`

## Опис логічного зв'язку
* **Use Case Diagram** задає структуру: ми бачимо, що Речовик вносить дані, а Система автоматично контролює строки.

* **Sequence Diagram** деталізує один з варіантів використання — "Внесення даних про видачу". Вона показує, як дані потрапляють у БД та як ініціюється механізм розрахунку часу.

* **Activity Diagram** розширює функцію "Автоматичне сповіщення та контроль". Вона описує життєвий цикл майна після того, як воно було внесено в базу (через Sequence Diagram), і до моменту його заміни.

* **Інструкція з використання:**
* Ви можете скопіювати цей код у Mermaid Live Editor, щоб отримати зображення високої якості (PNG/SVG).
